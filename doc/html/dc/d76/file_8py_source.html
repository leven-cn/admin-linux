4 '''@package _cookbook
5 File I/O Cookbook.
6 
7 By reading and writing only large chunks of data even when the user asks for a
8 single byte, **buffered I/O** is designed to hide any inefficiency in calling
9 and executing the operating system’s unbuffered I/O routines. The gain will
10 vary very much depending on the OS and the kind of I/O which is performed (for
11 example, on some contemporary OSes such as Linux, unbuffered disk I/O can be as
12 fast as buffered I/O). The bottom line, however, is that buffered I/O will
13 offer you predictable performance regardless of the platform and the backing
14 device. Therefore, _it is most always preferable to use buffered I/O rather
15 than unbuffered I/O_.
16 
17 The I/O system is built from layers. Text files are constructed by adding a
18 text encoding/decoding layer on top of a buffered binary-mode file. The
19 `buffer` attribute simply points at this underlying file. If you access it,
20 you’ll bypass the text encoding/decoding layer. You could write raw bytes to a
21 file opened in text mode using this technique.
22 
23 
24 For **text I/O**, reading line by line is more common.
25 
26  `for line in f`
27 
28 
29 For **binary I/O**,
30 
31  >>> test_bin_io('_file.data')

33  2 bytearray(b'cd')
34  2 bytearray(b'ef')
35  1 bytearray(b'Gf')
36  b'ab'

97 def test_mmap(filename):
98  '''Memory-mapped I/O.
99 
100  It should be emphasized that memory mapping a file does not cause the
101  entire file to be read into memory. That is, it’s not copied into some kind
102  of memory buffer or array. Instead, the operating system merely reserves a
103  section of virtual memory for the file contents. As you access different
104  regions, those portions of the file will be read and mapped into the memory
105  region as needed. However, parts of the file that are never accessed simply
106  stay on disk. This all happens transparently, behind the scenes.
107 
108  If more than one Python interpreter memory maps the same file, the
109  resulting mmap object can be used to exchange data between interpreters.
110  That is, all interpreters can read/write data simultaneously, and changes
111  made to the data in one interpreter will automatically appear in the
112  others. Obviously, some extra care is required to synchronize things, but
113  this kind of approach is sometimes used as an alternative to transmitting
114  data in messages over pipes or sockets.
115  '''

117  import os
