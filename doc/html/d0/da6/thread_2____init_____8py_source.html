7 The `threading` module constructs higher-level threading interfaces on top of

9 
10 **NOTE**: In CPython, due to the Global Interpreter Lock (GIL), only one
11 thread can execute Python code at once (even though certain
12 performance-oriented libraries might overcome this limitation).If you want
13 your application to make better use of the computational resources of
14 multi-core machines, you are advised to use multiprocessing. However,
15 threading is still an appropriate model if you want to run multiple I/O-bound
16 tasks simultaneously. In other words, I/O bound Python applicaitons stand a
17 much better chance of being able to take advantage of a multithreaded
18 environment than CPU-bound one.
19 
20 **NOTE**: While the `import` machinery is thread-safe, there are two key
21 restrictions on threaded imports due to inherent limitations in the way that
22 thread-safety is provided:
23 
24  - Firstly, other than in the main module, an `import` should not have the
25  side effect of spawning a new thread and then waiting for that thread in
26  any way. Failing to abide by this restriction can lead to a deadlock if
27  the spawned thread directly or indirectly attempts to import a module.
28 
29  - Secondly, all `import` attempts must be completed before the interpreter
30  starts shutting itself down. This can be most easily achieved by only
31  performing imports from non-daemon threads created through the threading
32  module. Daemon threads and threads created directly with the `thread`
33  module will require some other form of synchronization to ensure they do
34  not attempt imports after system shutdown has commenced. Failure to
35  abide by this restriction will lead to intermittent exceptions and
36  crashes during interpreter shutdown (as the late imports attempt to
37  access machinery which is no longer in a valid state).
